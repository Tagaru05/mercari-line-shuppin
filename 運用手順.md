# 運用手順（実行者＝このPCのClaude Code）— v0.3

先行環境で実走して確定した手順。**この順序と罠対策を必ず守る。**
Worker URLと鍵は `config.md` を読む。認証は必ず `Authorization: Bearer $(cat worker/.list_key)` ヘッダーで行い、**URLにキーを載せない**。

## 監視（このフォルダのセッション開始時に常駐）

```bash
KEY=$(cat worker/.list_key)
curl -s -H "Authorization: Bearer $KEY" "https://（WorkerURL）/list"   # {"keys":[...box/...]}
```
- Monitorで60秒間隔ポーリング。新着→即パイプライン実行。止めたい時はユーザーが「監視止めて」
- 200件を超える未処理は一度に見えない（通常ありえないが、超えたら分割処理）

## パイプライン（1商品・並行処理の型）

### ① 写真回収と商品認識
```bash
curl -s -H "Authorization: Bearer $KEY" "https://（URL）/get/（URLエンコードキー）" -o item.jpg
sips -Z 700 item.jpg --out item_check.jpg   # 目視は縮小版で（トークン節約）
```
- 写真をReadで確認し商品を特定。**食品・医薬品・偽ブランド疑い等は下書きを作らず、理由を添えてユーザーへ通知**
- ブランド品はブランド名・型番をユーザーに一言確認してから進める

### ② 先にGPTsへ投げる（生成待ちを並行時間にする）
- GPTs: https://chatgpt.com/g/g-6a5a283250588191bcc0fec46076a89b-merukarihua-xiang-sheng-cheng-turu （開けない時は `02_GPTs指示文（予備）.md` で自作）
- **画像の渡し方**：まず署名付き一時URLを発行（マスターキーをブラウザに出さないため）：
```bash
curl -s -H "Authorization: Bearer $KEY" "https://（URL）/sign?key=box%2F…jpg&ttl=600"   # → {"url": "..../public/...?exp=..&sig=.."}
```
- その一時URLを使い、ChatGPTページ内JSで `img(crossorigin=anonymous)→canvas→toBlob→File→ClipboardEvent('paste')` を composer に注入する。
  **注**：これは先行環境で確認した暫定経路（fetch/XHRはCSPで不可・クリップボード貼付は不安定）。**失敗したらユーザーに「この画像をChatGPTに貼ってください」と手動添付を案内**して先へ進む
- 依頼文の型：「この商品（○○）のメルカリ出品用画像をお願いします。商品そのものの形・色・状態は変えず、背景だけ明るく清潔感のある部屋風にしてください。」
- 送信したら**放置して③④へ**（生成1〜2分）

### ③ 白抜き（同梱スクリプト・複数パーツ対応）

### ④ 相場リサーチ（数値化した算式）
- 別タブで `jp.mercari.com/search?keyword=（商品名）&status=sold_out` を開く（GPTsタブは触らない）
- **売り切れ＝実売価格**のみ。同一・同類商品だけ抽出し、**5件以上**集める（5件未満ならキーワードを変えて再検索。それでも不足なら「相場データ薄」とユーザーに明記）
- 算式：**標準価格 = 実売の中央値 × 0.95 を10円単位に丸め**（下限300円）。強気=上位25%点、即売り=下位25%点も併記
- タイトル（40字）：`【状態語】+商品名+ブランド/型番+特徴語`
- 説明文の型：商品について／状態／写真について／発送＋ハッシュタグ4〜5個

### ⑤ GPTs回収
- `alt`が「生成された画像」のimgをページ内JSで `a.download` →Downloads→作業フォルダ→ `/upload/work%2Fitem_shohin.jpg` でKVへ
- **回収した生成画像を元写真と見比べ、商品の形・色・文字が変わっていないか確認**。変わっていたら不採用（白抜き＋原本のみで進める）

### ⑥ メルカリ下書き投入（罠対策込み）

**写真は3枚・この順**：**1枚目=元写真（未加工）／2枚目=商品風（加工）／3枚目=白抜き**
※規約安全側の既定。ユーザーが「加工画像を1枚目にしたい」と言った場合のみ入れ替える（完了報告で選べることを伝える）

1. jp.mercari.com/sell/create を開く
2. 写真注入：`/sign` で一時URLを発行→fetch→File×3→ input[type=file] に DataTransfer で注入
3. モーダル「次へ」→ AI出品サポートのカテゴリ候補から選択（無ければスキップ）
4. 商品名・価格を native setter＋input/changeイベントで投入
5. 配送select：shippingPayer=2(送料込み)／shippingFromArea=config.mdの都道府県コード／shippingDuration=1
6. **説明文は必ず最後に投入**（ページ遷移で消える罠・実証済み）
7. 説明文に必ず入れる：「2枚目は背景を加工したイメージ画像です。1枚目が実物の写真です」
8. 「下書きに保存する」→「下書きを保存しました」を実物確認
9. **出品するボタンは押さない**（最終判断は必ずユーザー）

### ⑦ 後処理
```bash
# box/→done/ へ移動（done/ は7日で自動削除される）
curl -s -X POST -H "Authorization: Bearer $KEY" -H 'content-type: application/json' \
  -d '{"keys":["box/…jpg"]}' "https://（URL）/done"
# 完了報告をLINEへ（Worker経由・トークンをローカルに持たない）
curl -s -X POST -H "Authorization: Bearer $KEY" -H 'content-type: application/json' \
  -d '{"text":"📦 出品準備が完了しました！\n（商品名）（価格）円\n\n下書き一覧を開いて、確認→出品ボタンだけお願いします。\n※1枚目は実物写真です。2枚目の加工画像を表紙にしたい場合は下書きで並べ替えできます"}' \
  "https://（URL）/notify"
```

### メモの紐づけ規則
- LINEのテキストメモは**直前・直後5分以内の写真と同じ商品**として扱う
- 複数商品を続けて送る時は、ユーザーに「商品ごとに『次』と送ってもらう」区切り運用を案内
- 解釈に迷ったら完了報告で「メモは○○と解釈しました」と明記

## 守る線（全商品共通）

1. 出品ボタンはユーザー。AIは絶対に押さない
2. 実物と違って見える加工はしない（傷消し・色盛りNG。背景と明るさのみ）。生成画像は毎回元写真と照合し、変わっていたら不採用
3. **1枚目は元写真（未加工）が既定**。説明文に加工の明記
4. フリマサービスの規約・仕様は変わりうる。**このキットは規約準拠を保証しない（利用は自己責任）**。挙動がおかしい時・警告が出た時は自動処理を止め、手動運用に切り替えてユーザーに報告する
5. 出品禁止・グレーの物（食品・医薬品・チケット・ブランド真贋不明など）は下書きを作らない

## データの扱い（ユーザーに説明できるように）

- 写真は受付サーバーに一時保存され、**未処理30日・処理済み7日で自動削除**される
- 全部すぐ消したい時：`wrangler kv key list/delete` で削除できる（あなたが代行する）

## トークン節約

- スクショは要所のみ。状態確認はDOM読みを優先／写真の目視は縮小版／複数商品はまとめて処理
